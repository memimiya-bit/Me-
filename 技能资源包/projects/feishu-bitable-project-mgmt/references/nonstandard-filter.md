## 为什么需要技能系统

### 静态过滤的问题

| 问题 | 说明 | 示例 |
|-----|------|------|
| 业务差异 | 同一词在不同部门含义不同 | "接口"=技术API / 销售客户对接人 |
| 动态识别 | 简单计数会误伤关键业务词 | "合同"在法务群高频=正常 |
| 维护成本 | 硬编码规则难以扩展 | 新场景需改代码 |

### 技能系统优势

| 优势 | 说明 |
|-----|------|
| 避免硬编码 | 规则变更只需更新技能，无需改主流程 |
| 场景隔离 | 销售、法务等不同技能独立维护 |
| 动态适应 | 可根据上下文动态调整规则 |
| 可复用 | 同类场景可直接复制技能 |

```
┌─────────────────────────────────────────────────────────────┐
│                      技能调度层                             │
│                   （场景识别+技能路由）                       │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  销售场景技能 │    │  法务场景技能 │    │  技术场景技能 │
│  非标词过滤   │    │  非标词过滤   │    │  非标词过滤   │
└───────────────┘    └───────────────┘    └───────────────┘
        │                     │                     │
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  销售术语库   │    │  法务术语库   │    │  技术术语库   │
└───────────────┘    └───────────────┘    └───────────────┘
```

### 技能结构

```yaml
技能名称: 非标词过滤_销售场景
版本: 1.0
触发条件: 消息含销售相关字段（客户/金额/合同）
规则集:
  - 规则ID: RULE_001
    名称: 高频词识别
    条件: 词频 > 10 且 不在销售术语库
    动作: 标记为"疑似非标词"
  - 规则ID: RULE_002
    名称: 语义验证
    条件: 已标记为"疑似非标词"
    动作: 调用语义模型验证
    验证规则: 是否指代标准术语（如"客资"=客户资料）
  - 规则ID: RULE_003
    名称: 处理决策
    条件: 语义验证完成
    动作: >
      置信度 > 85%: 自动替换
      置信度 30-85%: 降低权重
      置信度 < 30%: 转人工确认
```

## 场景化技能示例

### 技能1：销售场景非标词过滤

```yaml
技能名称: 非标词过滤_销售场景
版本: 1.0

触发条件:
  OR(
    CONTAINS(消息, {"客户", "订单", "合同", "金额"}),
    CONTAINS(消息, {"跟进", "回款", "报价"}),
    当前群组类型 = "销售部"
  )

过滤规则:
  # 规则1：高频非标准词识别
  - ID: RULE_SALES_001
    条件: >
      AND(
        词频 > 10,
        NOT(术语库匹配("销售术语库", 词汇))
      )
    动作: 标记为"疑似非标词"
    
  # 规则2：语义验证
  - ID: RULE_SALES_002
    条件: 已标记为"疑似非标词"
    验证提示: "判断{词汇}是否指代标准销售术语"
    标准术语库:
      - "客资" → "客户资料"
      - "订单" → "销售订单"
      - "回款" → "应收账款"
      
  # 规则3：处理决策
  - ID: RULE_SALES_003
    条件: 语义验证完成
    分支:
      - 置信度 > 85%: 自动替换为标准术语
      - 置信度 30-85%: 降低权重至0.3，提示用户
      - 置信度 < 30%: 忽略，加入白名单

输出:
  处理状态: "已处理" / "待确认" / "已忽略"
  标准术语: 替换后的标准词汇
  置信度: 0-100
```

### 技能2：技术场景非标词过滤

```yaml
技能名称: 非标词过滤_技术场景
版本: 1.0

触发条件:
  OR(
    CONTAINS(消息, {"接口", "API", "服务", "开发"}),
    CONTAINS(消息, {"部署", "测试", "Bug"}),
    当前群组类型 = "技术部"
  )

过滤规则:
  # 规则1：部门特定含义识别
  - ID: RULE_TECH_001
    条件: >
      OR(
        词汇 = "接口",
        词汇 = "对接"
      )
    处理: >
      IF(
        当前部门 = "技术部",
        "API接口（系统对接）",
        "客户对接人（人际沟通）"
      )
      
  # 规则2：缩写识别
  - ID: RULE_TECH_002
    条件: >
      AND(
        词频 > 5,
        词长度 <= 5,
        NOT(术语库匹配("技术术语库", 词汇))
      )
    动作: 语义验证是否为技术缩写

标准术语映射:
  "工单" → "技术支持工单"
  "需求" → "功能需求"
  "Bug" → "缺陷"
  "API" → "应用程序接口"

输出:
  处理状态: "已处理" / "待确认" / "已忽略"
  部门语义: 该词汇在当前部门的含义
  置信度: 0-100
```

### 技能3：法务场景非标词过滤

```yaml
技能名称: 非标词过滤_法务场景
版本: 1.0

触发条件:
  OR(
    CONTAINS(消息, {"合同", "协议", "条款"}),
    CONTAINS(消息, {"法务", "审核", "合规"}),
    当前群组类型 = "法务部"
  )

过滤规则:
  # 规则1：法律术语标准化
  - ID: RULE_LAW_001
    条件: 词汇在法律术语变体库中
    动作: 替换为标准法律术语
    
  # 规则2：非标准表达识别
  - ID: RULE_LAW_002
    条件: >
      AND(
        词频 > 3,
        NOT(术语库匹配("法务术语库", 词汇)),
        词汇 IN {"协议", "约定", "承诺"}
      )
    动作: 提示使用更精确的法律术语

标准术语映射:
  "协议" → "合同（民事法律行为）"
  "约定" → "合同条款"
  "承诺" → "要约/承诺"

输出:
  处理状态: "已处理" / "待确认"
  建议术语: 更精确的法律术语
  置信度: 0-100
```

## 实施配置

### 飞书自动化配置

**多维表格字段**：

| 字段名 | 类型 | 公式 |
|-------|------|------|
| 非标词识别 | 单选 | 技能执行结果 |
| 标准术语 | 单行文本 | 替换建议 |
| 处理状态 | 单选 | 自动/待确认/忽略 |
| 置信度 | 数字 | 0-100 |

**自动化规则**：

```formula
// 技能调度规则
技能触发 = SWITCH(
  TRUE(),
  OR(销售触发条件), "非标词过滤_销售场景",
  OR(技术触发条件), "非标词过滤_技术场景",
  OR(法务触发条件), "非标词过滤_法务场景",
  "默认过滤"
)

// 自动处理
处理结果 = IF(
  置信度 > 85,
  自动替换(非标词, 标准术语),
  IF(置信度 > 30, 降低权重, 忽略)
)
```

## 效果评估

### 核心指标

| 指标 | 目标值 | 说明 |
|-----|-------|------|
| 识别准确率 | >90% | 正确识别的非标词/总识别数 |
| 召回率 | >85% | 正确识别/应识别的非标词 |
| 自动处理率 | >80% | 无需人工介入的处理比例 |
| 用户满意度 | >85% | 替换建议被采纳的比例 |

### 评估公式

```formula
// 识别准确率
准确率 = 正确识别数 / (正确识别数 + 误判数) * 100%

// 召回率
召回率 = 正确识别数 / (正确识别数 + 漏判数) * 100%

// 综合评分
综合评分 = 准确率 * 0.5 + 召回率 * 0.3 + 自动处理率 * 0.2
```

### 持续优化机制

1. **周度复盘**：分析误判案例，更新规则
2. **月度评估**：整体效果评估，调整策略
3. **季度优化**：全面审查，版本升级
