## 修正指令机制

### 修正触发方式

```
当你发现生成的指令不准确时，回复：
#修正 规则名

示例：
#修正 合同审核
#修正 库存预警
```

### 修正输入格式

```
#修正 规则名
修正原因：{你输入的文本}

示例：
#修正 合同审核
修正原因：合同版本号应该提取"v3.0"格式，不是"3.0"
```

## 修正记录管理

### 「规则修正记录」表结构

| 字段名 | 类型 | 说明 |
|-------|------|------|
| 规则名称 | 单行文本 | 被修正的规则 |
| 原始消息 | 多行文本 | 用户发送的消息 |
| 生成结果 | 多行文本 | 系统生成的指令 |
| 修正结果 | 多行文本 | 用户手动修改后的结果 |
| 修正原因 | 多行文本 | 用户输入的修正原因 |
| 修正时间 | 日期时间 | 自动记录 |
| 提取字段 | 多行文本 | 从消息中提取的字段 |
| 状态 | 单选 | 待确认/已优化/忽略 |

## 每周修正报告

### 自动生成机制

每周一自动生成「规则修正报告」：

```
【规则修正报告】- 2025年5月第1周

一、修正统计
- 本周总触发次数：156次
- 本周修正次数：5次
- 修正率：3.2%（目标<5%）

二、修正明细
| 规则名 | 修正次数 | 主要原因 |
|--------|----------|---------|
| 合同审核 | 3次 | 合同版本号格式不一致 |
| 库存预警 | 2次 | 阈值判断有误 |

三、优化建议
根据修正记录分析：
1. 合同审核规则中，"版本号"提取正则建议更新为：
   匹配 "v?([\d.]+)" 同时支持"3.0"和"v3.0"
   
2. 库存预警规则中，安全阈值应按品类分别设置

四、待确认优化
请确认以下优化是否正确：
[ ] 更新合同版本号提取正则
[ ] 分品类设置安全库存阈值

确认时间：本周五前
```

### 确认流程

```
生成报告 → 发送给你 → 你确认优化项 → 执行优化 → 规则更新
     ↑                                                    │
     └──────────────── 5分钟确认 ─────────────────────────┘
```

## 规则自动优化

### 基于修正的自动优化规则

```formula
// 自动权重调整
如果某修正原因涉及特定字段，则提升该字段的提取权重
修正原因包含"版本号" → 提升"版本号"提取权重 +10%

// 自动正则优化
如果3次以上修正涉及同一字段格式，则建议更新正则
修正"版本号"格式>3次 → 建议更新正则为更通用格式

// 自动规则拆分
如果修正涉及不同场景但同一规则，则建议拆分
同一规则在"合同审核"和"采购合同"场景修正 → 建议拆分为独立规则
```

### 优化触发条件

| 触发条件 | 优化动作 |
|---------|---------|
| 同一字段修正>3次 | 更新该字段提取正则 |
| 同一规则修正>5次 | 触发深度复盘 |
| 修正原因涉及上下文 | 补充触发条件 |
| 修正涉及多场景 | 建议规则拆分 |

## 关键指标验证

### 核心指标

| 指标 | 目标值 | 测量方式 | 预警阈值 |
|-----|-------|---------|---------|
| 人工复核率 | <15% | 需确认任务数/总触发数 | >20% |
| 规则修正率 | <5% | 修正次数/总使用次数 | >8% |
| 单任务耗时 | ≤原流程50% | 自动化前后对比 | 无改善 |
| 规则满意度 | >80% | 用户确认执行率 | <70% |

### 指标计算公式

```formula
// 人工复核率
人工复核率 = 需确认任务数 / 总触发任务数 * 100%

// 规则修正率
规则修正率 = 修正次数 / 规则总使用次数 * 100%

// 单任务耗时节省
耗时节省 = (原流程耗时 - 自动化耗时) / 原流程耗时 * 100%
```

### 指标异常处理

```
若人工复核率 > 15%：
  1. 立即停用该规则
  2. 分析修正记录，找出误触发原因
  3. 用Excel法提取真实触发条件
  4. 更新触发条件后重新启用

若规则修正率 > 5%：
  1. 深度复盘修正记录
  2. 检查字段提取是否准确
  3. 补充遗漏的上下文条件
  4. 必要时拆分规则

若单任务耗时无改善：
  1. 排查自动化流程瓶颈
  2. 简化操作步骤
  3. 考虑是否适合自动化
```

## 为什么比AI学习可靠

### AI学习的痛点

| 问题 | 示例 |
|-----|------|
| 通用模型误判 | "学习合同法"被识别为工作指令 |
| 过度泛化 | 将正常讨论识别为需要处理的任务 |
| 数据依赖 | 需要大量标注数据，积累周期长 |
| 不可解释 | 无法说明为何做出某个判断 |

### 人工反馈的优势

| 优势 | 说明 |
|-----|------|
| 明确业务场景 | 修正记录明确指向"合同审核"而非泛化 |
| 可解释性 | 修正原因清晰记录，无需猜测 |
| 快速迭代 | 每周5分钟确认即可更新规则 |
| 精准控制 | 所有操作需人工确认，100%可控 |

### 自动学习规则

```
当修正记录出现规律时，系统自动学习：

规律1：3次修正都涉及"版本号"
→ 自动提升"合同"+"版本号"组合的权重

规律2：修正原因都涉及"格式"
→ 自动建议更新提取正则

规律3：修正分布在不同场景
→ 自动建议规则拆分
```

## 实施检查清单

### 启动阶段

```
□ 创建「操作模板库」表
□ 创建「规则修正记录」表
□ 配置第一个自动化规则（建议选高频场景）
□ 测试规则触发和确认流程
□ 培训团队使用 #修正 指令
```

### 运行阶段

```
□ 每日检查待办确认
□ 每周一查看修正报告
□ 每周五前确认优化建议
□ 每月复盘规则效果
□ 持续补充新模板
```

### 优化阶段

```
□ 分析高频修正原因
□ 优化触发条件和字段提取
□ 拆分复杂规则
□ 扩展模板覆盖场景
```
