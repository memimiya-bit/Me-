## 核心理念

**痛点**：企业知识散落各处，员工平均每天浪费28分钟寻找关联文档。

**目标**：建立文档智能关联网络，让知识主动找到需要它的人。

## 关联强度计算

### 多维度关联公式

```formula
关联强度 = (
  {文本语义相似度} × 0.4 + 
  {共同引用次数} × 0.3 + 
  {作者协作频次} × 0.15 + 
  {时间临近度} × 0.15
) × 100
```

### 各维度计算方式

| 维度 | 权重 | 计算方式 | 数据来源 |
|-----|-----|---------|---------|
| 文本语义相似度 | 40% | AI语义分析/关键词重叠 | 文档内容 |
| 共同引用次数 | 30% | 引用同一文档的次数 | 引用关系 |
| 作者协作频次 | 15% | 共同作者项目数 | 协作记录 |
| 时间临近度 | 15% | 发布时间相近程度 | 元数据 |

### 关联强度等级

| 关联强度 | 等级 | 展示方式 |
|---------|-----|---------|
| >=80% | 强关联 | 实线+放大镜图标 |
| 50-79% | 中关联 | 虚线+链接图标 |
| 20-49% | 弱关联 | 细虚线+提示图标 |
| <20% | 无关 | 不展示 |

## 关联发现机制

### 自动关联发现

```formula
关联发现 = AI_ANALYZE(
  {当前文档} + {知识库},
  "分析文档间的语义关联，找出与'{当前文档}'相关的文档，
   标注关联类型（引用/继承/参考/相关），用'商业语言'说明关联价值"
)
```

### 关联类型识别

| 类型 | 标识 | 说明 | 示例 |
|-----|-----|------|-----|
| 引用(Reference) | → | 直接引用数据/结论 | 需求文档引用用户调研 |
| 继承(Extend) | ↗ | 基于原文档扩展 | V2方案继承V1方案 |
| 参考(Guide) | ⤴ | 作为执行指导 | 开发文档参考架构设计 |
| 相关(Related) | ↔ | 业务上存在关联 | 同项目不同阶段 |

### 智能推荐生成

```formula
推荐理由 = IF(
  {关联类型} = "强关联",
  "关键发现：'{引用内容}'（引用自{文档名称}，{时间}）",
  IF({关联类型} = "中关联",
    "相关模块：'{核心概念}'（见{文档名称}第{章节}章）",
    "参考信息：'{相关点}'（详见{文档名称}）"
  )
)
```

```
                    [需求文档：支付优化]
                          │
          ┌───────────────┼───────────────┐
          │               │               │
          ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐

    └──────────┘    └──────────┘    └──────────┘
        │               │               │
     强关联          中关联          弱关联
       85%             65%             35%
        │               │               │
        ▼               ▼               ▼
   "62%用户因       "异步支付        "支付失败
    支付失败流失"    状态机"         话术模板"
```

### 关键发现展示

**强关联(>=80%)**：
```
[用户调研报告2024Q3]
引用内容："62%用户因支付失败流失"
关联价值："此数据是本次支付优化的核心依据，
          优化方案应重点解决支付失败场景"
建议行动："在测试用例中增加支付失败场景覆盖"
```

**中关联(50-79%)**：
```
[技术方案V2.1]
核心模块："异步支付状态机"
关联价值："技术实现需考虑与现有状态机的兼容性"
建议参考："第3.2章节详细说明了状态转换逻辑"
```

**弱关联(20-49%)**：
```
[客服培训手册]
相关策略："支付失败话术模板"
关联价值："优化上线后需同步更新客服培训内容"
建议时机："优化上线前1周通知客服团队"
```

## 知识孤岛治理

### 孤岛识别标准

```formula
孤岛指数 = (
  {被引用次数} = 0 ? 100 : 0 +
  {被引用次数} < 3 ? 50 : 0 +
  {更新时间} < 180天 ? 30 : 0
)
```

### 孤岛类型

| 类型 | 特征 | 治理策略 |
|-----|------|---------|
| 沉睡文档 | 长期未更新，无人访问 | 归档或删除 |
| 孤儿文档 | 无引用关系，孤立存在 | 建立关联或合并 |
| 重复文档 | 内容高度相似 | 合并或建立主从 |
| 过期文档 | 信息已过时 | 标记过期，引用新文档 |

### 智能推荐场景

**场景1：新建需求文档时**
```
系统自动推荐：
├── 相关历史需求：{列表}
│   └── 复用建议：可直接引用{历史文档}的需求分析
├── 参考技术文档：{列表}
│   └── 开发团队可参考{技术文档}的接口设计
└── 关联用户研究：{列表}
    └── 建议引用{调研报告}的用户反馈数据
```

**场景2：项目复盘时**
```
系统自动推荐：
├── 本项目历史文档：{列表}
│   └── 建议按{模板}格式整理复盘报告
├── 类似项目参考：{列表}
│   └── {项目名称}的复盘经验可借鉴
└── 相关教训总结：{列表}
    └── {教训文档}记录的问题需避免
```

**场景3：知识搜索时**
```
搜索关键词："支付优化"
智能扩展：
├── 同义词：付款优化、交易优化
├── 相关词：用户流失、转化率
└── 互补词：客服培训、风控策略

推荐文档（按关联度排序）：
1. [用户调研报告2024Q3] - 85%相关
2. [支付功能PRD V2.3] - 78%相关
3. [技术方案V2.1] - 65%相关
4. [客服培训手册] - 35%相关
```

## 实施配置

### 文档分类体系

| 类别 | 标签 | 核心规则 |
|-----|-----|---------|
| 战略文档 | strategy | 1年内有效，高管审批 |
| 需求文档 | requirement | 版本管理，关联设计 |
| 技术文档 | technical | 代码同步，架构评审 |
| 运营文档 | operation | 定期更新，责任到人 |
| 流程文档 | process | BPMN标准，定期审计 |

### 关联更新策略

```formula
更新触发 = OR(
  {文档内容变更},
  {新增引用关系},
  {时间达到定期周期}
)

更新频率 = SWITCH({文档类型},
  "战略文档", "月度",
  "需求文档", "实时",
  "技术文档", "周度",
  "运营文档", "周度",
  "流程文档", "季度"
)
```

### 质量评分

```formula
文档健康度 = (
  关联文档数 × 10 +    // 每关联一个文档+10
  更新及时性 × 30 +    // 按更新周期评分
  内容完整度 × 30 +    // 模板完整度
  引用准确度 × 30     // 被正确引用次数
)
```
